# TwitterRX Dashboard コンスティテューション

## コア原則

### I. クリーンコード（交渉不可）

すべての実装コードは保守性を最優先とする。\
以下のルールを常に遵守しなければならない（MUST）:

- 意図が自明な命名を採用すること（省略形・単一文字変数名は原則禁止）。
- 関数・メソッドは副作用を最小化し、戻り値の型を明示すること。
- 魔法の数値・文字列はすべて定数または Enum として抽出すること。
- コードの重複は即座に抽象化し、同じロジックを2箇所以上に書かないこと（DRY）。

**根拠**: Astro SSR + Node.js という構成では、サーバーサイドとクライアントサイドの
境界が曖昧になりやすい。コードの可読性はバグ混入リスクと直結するため、
クリーンコードは長期的な開発速度の根幹である。

---

### II. 早期リターン（ガード節）

条件分岐においては、早期リターン（ガード節）パターンを必ず採用しなければならない（MUST）。

- 正常系フローを関数末尾に1つだけ置き、エラー・境界条件を先頭でガードすること。
- ネストが2段以上になるケースは早期リターンに書き直すこと。
- `else` ブロックは早期リターン後に残すことを禁止する。

```typescript
// NG
function getConfig(guildId: string | null) {
  if (guildId !== null) {
    if (isValidSnowflake(guildId)) {
      return db.query(guildId);
    }
  }
}

// OK
function getConfig(guildId: string | null): Promise<GuildConfig | null> {
  if (guildId === null) return Promise.resolve(null);
  if (!isValidSnowflake(guildId)) return Promise.resolve(null);
  return db.query(guildId);
}
```

**根拠**:
早期リターンはネストの深さを自然に抑制し、テストケースの可視性を高める。

---

### III. テスト容易性ファースト

すべてのモジュールはテストコードが書きやすい設計でなければならない（MUST）。

- 依存関係は引数注入（DI）で渡し、モジュール内部でのハードコードを禁止する。
- Discord API・Redis など副作用を持つ外部依存への直接呼び出しは、薄いアダプター層
  （`DiscordClient`・`RedisClient` など）を介してのみ行うこと。
- テスト対象の関数は純粋関数を優先し、グローバル状態への依存を最小化すること。
- Vitest を用いたユニットテストを各ドメインモジュール（`src/lib/`）に対して作成すること。

**根拠**:
認証・セッション・暗号化などのセキュリティ機能は実装が複雑で回帰しやすい。
テスト可能な設計は早期にリグレッションを検出し、リファクタリングを安全に行う土台を提供する。

---

### IV. メソッドサイズ制限

関数・メソッドの行数は原則 **30行以内** とする。\
複雑な処理（データ変換・クエリ構築・暗号化ロジック等）は **最大60行** まで認める。

- 30行を超える場合は、コードレビュー時にレビュアーへ超過理由を明示すること（MUST）。
- 60行を超えることは、いかなる理由があっても許可しない（MUST NOT）。
- 行数制限を超えそうな場合、まず責務の分割（原則V）を検討すること。

**根拠**: 認知負荷の上限を設けることで、コードレビューの質と速度を保つ。
行数制限は設計の問題を早期に顕在化させるセーフガードでもある。

---

### V. 単一責任原則（SRP）

クラス・関数・モジュールはそれぞれ **1つの責務のみ**
を持たなければならない（MUST）。

- UI コンポーネント（Astro/Preact）は描画のみを責務とし、ビジネスロジックを含めない。
- API ルート（`src/pages/api/`）はリクエスト/レスポンス処理のみを担い、
  ドメインロジックを直接実装しない。
- ビジネスロジック（認証・レート制限・暗号化など）は `src/lib/` に閉じること。
- DB アクセスは `src/lib/db/` に集約し、それ以外の層から直接クエリを発行しない。
- ファイル名はその責務を端的に表すこと（例: `auth.ts`, `rate-limit.ts`, `crypto.ts`）。

**根拠**: SRP への違反は変更影響範囲を拡大し、テスト設計を困難にする。
Web ダッシュボードでは UI・API・ビジネスロジック・DB が密結合しやすいため、
境界を明確にコンスティテューションで定義する。

## 技術スタック

| 役割               | 技術                                                  |
| ------------------ | ----------------------------------------------------- |
| 言語               | TypeScript（strict モード、`any` 型の使用禁止）       |
| ランタイム         | Node.js（@astrojs/node アダプター経由）               |
| Web フレームワーク | Astro v4（SSR/server モード）                         |
| フロントエンド UI  | Preact                                                |
| データベース       | SQLite（better-sqlite3）+ Drizzle ORM                 |
| セッション / KV    | Redis（ioredis）                                      |
| 認証               | Discord OAuth2（arctic）、oslo、CSRF 保護             |
| テスト             | Vitest                                                |
| Linting            | oxlint                                                |
| フォーマット       | oxfmt                                                 |
| ターゲット         | サーバーサイド（Node.js）+ ブラウザ（モダンブラウザ） |

**制約**:

- `any` 型の使用を禁止する（MUST NOT）。型推論が困難な場合は `unknown` +
  型ガードを使うこと。
- Drizzle ORM を介さずに SQLite へ直接クエリを発行してはならない（MUST NOT）。
- セッション操作は必ず `src/lib/auth.ts` を経由すること。
  Astro ルートで lucia/oslo のAPIを直接呼び出すことを禁止する。

---

## コード品質ゲート

すべての PR はマージ前に以下のゲートをパスしなければならない（MUST）:

1. **Linting**: `npm run lint` がエラーなしで通過すること。
2. **フォーマット**: `npm run format:check` がエラーなしで通過すること。
3. **型チェック**: `npm run build`（`astro check` を含む）がエラーなしで通過すること。
4. **ユニットテスト**: `npm run test` で全テストがグリーンであること。
5. **メソッドサイズ**:
   60行を超える関数が存在しないこと（レビュアーが目視確認）。
6. **SRP チェック**: UI / API ルート / ビジネスロジック / DB の責務が混在していないこと。

## ガバナンス

このコンスティテューションは他のすべての開発プラクティスに優先する。\
修正を行う場合は以下の手順を踏むこと:

1. 変更内容と理由を PR の説明に明記する。
2. `CONSTITUTION_VERSION`
   をセマンティックバージョニングに従ってインクリメントする。
3. `LAST_AMENDED_DATE` を更新する。
4. 依存するテンプレート（plan / spec /
   tasks）への影響を確認し、必要なら同時更新する。

すべてのコードレビューにおいて、レビュアーはコンスティテューションへの準拠を確認する責任を持つ。
