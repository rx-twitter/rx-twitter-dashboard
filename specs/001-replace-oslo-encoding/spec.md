# フィーチャー仕様: oslo/encoding を自前 Base32 実装に置き換え

**フィーチャーブランチ**: `001-replace-oslo-encoding`\
**作成日**: 2026-04-05\
**ステータス**: ドラフト\
**入力**: ユーザー説明: "oslo/encoding パッケージを自前の Base32 エンコーディング実装に置き換える。oslo パッケージがアーカイブされたため、RFC 4648 準拠の自前実装でセッション ID 生成の Base32 エンコードを行う。既存セッションとの互換性を維持しつつ外部依存を削除する。"

## ユーザーシナリオとテスト _（必須）_

### ユーザーストーリー1 - 既存セッション互換を保った oslo 依存の除去（優先度: P1）

運用チームは、アーカイブ済み `oslo` パッケージへの依存を除去し、
将来のセキュリティリスクやメンテナンス負担を排除したい。
置き換え後も、本番環境で稼働中の既存セッション（Redis に保存済みの Base32 形式セッション ID）が
引き続き有効であること。

**この優先度の理由**: アーカイブ済みパッケージは脆弱性パッチが提供されないリスクがある。
セッション認証はアプリケーションのセキュリティ根幹であり、最優先で対処が必要。

**独立テスト**: 自前 Base32 エンコード関数の単体テスト（既知入力→既知出力）と、
`generateSessionId()` から生成される ID の形式・一意性テストで完全に検証可能。
既存セッションへの影響は、エンコーディング方式が同一であることの仕様テストで担保される。

**受け入れシナリオ**:

1. **前提** oslo パッケージの import が存在しない状態で、**操作** セッション ID を生成する、**期待** RFC 4648 Base32 形式（A-Z, 2-7 のみ、32文字）の文字列が返される
2. **前提** 自前 Base32 実装が導入された状態で、**操作** 同一のバイト列を oslo の Base32 と自前実装でそれぞれエンコードする、**期待** 出力が一致する
3. **前提** 既存セッション ID が Redis に保存されている状態で、**操作** ユーザーがリクエストを送信する、**期待** セッションが正常に検証される（再ログイン不要）
4. **前提** 自前実装が導入された状態で、**操作** `npm install` を実行する、**期待** `oslo` が依存ツリーに含まれない

---

### ユーザーストーリー2 - Base32 エンコードのテストカバレッジ確保（優先度: P2）

開発者は、自前実装の正確性を継続的に保証するため、
エッジケースを含むテストスイートを保持したい。

**この優先度の理由**: 自前実装は外部ライブラリのテストに依存できないため、
独自のテストカバレッジが品質保証の唯一の手段となる。

**独立テスト**: Base32 エンコード関数に対する単体テスト（空入力、1バイト、各種長さ、既知ベクター）
で独立して検証可能。

**受け入れシナリオ**:

1. **前提** 空のバイト列、**操作** Base32 エンコードを実行する、**期待** 空文字列が返される
2. **前提** 1バイト（0x00）のバイト列、**操作** Base32 エンコードを実行する、**期待** RFC 4648 に準拠した出力が返される
3. **前提** 20バイトのランダムバイト列、**操作** Base32 エンコードを実行する、**期待** A-Z, 2-7 のみで構成された32文字の文字列が返される
4. **前提** RFC 4648 のテストベクター、**操作** エンコードを実行する、**期待** 仕様通りの出力が返される

## 要件 _（必須）_

### 機能要件

- **FR-001**: システムは RFC 4648 Base32 アルファベット（A-Z, 2-7）に準拠した、パディングなし・任意バイト長のエンコード関数を `src/lib/encoding.ts` として提供しなければならない（MUST）。シグネチャ: `encodeBase32(bytes: Uint8Array): string`
- **FR-002**: 自前 Base32 エンコード関数は `oslo/encoding` の `base32.encode()` と同一の出力を生成しなければならない（MUST）
- **FR-003**: `package.json` から `oslo` 依存を完全に削除しなければならない（MUST）
- **FR-004**: `generateSessionId()` は 20バイト（160ビット）の暗号論的乱数を Base32 エンコードした文字列を返さなければならない（MUST）
- **FR-005**: 自前 Base32 エンコード関数に対するユニットテストを作成しなければならない（MUST）
- **FR-006**: 既存の `generateSessionId()` のテストは引き続きパスしなければならない（MUST）

| 原則               | このフィーチャーへの影響                                                                             |
| ------------------ | ---------------------------------------------------------------------------------------------------- |
| I. クリーンコード  | Base32 アルファベットは定数として定義。関数名は `encodeBase32` 等、意図が自明な命名とする            |
| II. 早期リターン   | 空入力チェックをガード節で先頭に配置                                                                 |
| III. テスト容易性  | Base32 エンコードは純粋関数として実装し、外部依存なしでテスト可能とする                              |
| IV. メソッドサイズ | エンコード関数は10行程度で収まり、30行制限を大幅に下回る                                             |
| V. SRP             | Base32 エンコードは `src/lib/encoding.ts` として独立モジュール化。`auth.ts` はそれを import するのみ |

## 成功基準 _（必須）_

### 測定可能な成果

- **SC-001**: `oslo` パッケージが `package.json` および `node_modules` から完全に除去されている
- **SC-002**: 既存のすべてのテスト（`npm run test`）が変更前と同一の結果でパスする
- **SC-003**: 自前 Base32 実装のテストカバレッジが RFC 4648 テストベクターを含む100%である
- **SC-004**: 本番環境のデプロイ後、既存ユーザーのセッションが中断なく継続される（再ログイン発生率 0%）

## 仮定

- `oslo/encoding` の `base32.encode()` は RFC 4648 Base32（パディングあり）だが、20バイト入力（160ビット）は5バイトブロック（40ビット）の倍数に揃うため `=` パディングは発生しない。自前実装は 20バイト入力で同一の32文字出力を生成すれば十分
- セッション ID は生成後 Base32 デコードされることがなく、文字列としてのみ使用される
- Redis に保存済みの既存セッション ID は TTL（7日間）による自然失効までそのまま有効であり、マイグレーションは不要
